# 本地图片资源浏览模板设计

## 背景

本项目将在当前目录初始化为 Electron、Vue 3、TypeScript 应用。第一版用于验证本地目录图片读取能力，为后续 Steam 游戏封面识别和展示功能打基础。

目标路径示例：

```text
C:\Program Files (x86)\Steam\appcache\librarycache
```

## 目标

第一版实现一个最小可用模板：

- 用户手动输入本地目录路径。
- 用户点击“读取图片”。
- Electron 主进程读取该目录下一级文件。
- 应用筛选常见图片文件。
- Vue 页面以网格展示图片预览、文件名、文件大小。
- 页面展示加载状态、错误状态、空状态和图片总数。

## 非目标

第一版不实现以下功能：

- 递归扫描子目录。
- 系统文件夹选择器。
- Steam AppID 识别。
- Steam 游戏名称匹配。
- 网络封面下载。
- 图片缓存管理。
- 图片编辑、删除或移动。

## 技术栈

- Electron：提供桌面应用外壳和本地文件系统访问。
- Vue 3：构建渲染进程界面。
- TypeScript：约束主进程、preload、渲染进程的数据结构。
- Vite：构建 Vue 渲染进程，并辅助 Electron 开发流程。

## 架构

应用分为三层。

### Electron 主进程

主进程负责访问本地文件系统，并提供 IPC 接口：

```ts
scanImages(directoryPath: string): Promise<ScanImagesResult>
```

主进程执行以下步骤：

1. 校验路径是否为空。
2. 检查路径是否存在。
3. 检查路径是否为目录。
4. 读取目录下一级文件。
5. 筛选图片扩展名。
6. 获取每张图片的文件大小。
7. 生成可用于渲染进程展示的 `file://` URL。
8. 返回图片列表。

支持的图片扩展名：

```text
.jpg, .jpeg, .png, .webp, .gif, .bmp
```

### Electron preload

preload 只暴露受控 API，不暴露 Node.js 原生模块：

```ts
window.imageLibrary.scanImages(directoryPath)
```

这样渲染进程无法直接访问 `fs` 或 `path`，但可以通过受控 IPC 获取图片数据。

### Vue 渲染进程

渲染进程负责界面和状态管理：

- 路径输入框默认填入 Steam librarycache 示例路径。
- “读取图片”按钮触发扫描。
- 扫描中禁用按钮并显示加载状态。
- 扫描成功后显示图片数量和图片网格。
- 扫描失败后显示错误信息。
- 扫描成功但无图片时显示空状态。

## 数据结构

```ts
interface ImageAsset {
  name: string
  absolutePath: string
  fileUrl: string
  extension: string
  sizeBytes: number
}

interface ScanImagesResult {
  images: ImageAsset[]
}
```

错误通过 IPC 抛出，渲染进程捕获后转成用户可读提示。

## 错误处理

应用处理以下情况：

- 路径为空：提示“请输入目录路径”。
- 目录不存在：提示“目录不存在”。
- 路径不是文件夹：提示“路径不是文件夹”。
- 无权限读取：提示“无法读取目录，请检查权限”。
- 目录中没有图片：显示空状态“该目录下没有可展示的图片”。

## 界面设计

页面采用简单单页布局：

1. 顶部标题：本地图片资源浏览器。
2. 路径输入区域：输入框和“读取图片”按钮。
3. 状态区域：显示加载、错误、空状态或图片数量。
4. 图片网格：每张卡片显示图片预览、文件名和文件大小。

第一版重视可用性，不追求复杂视觉效果。

## 测试与验证

实现后至少验证以下场景：

1. 输入不存在的路径，页面显示“目录不存在”。
2. 输入文件路径，页面显示“路径不是文件夹”。
3. 输入空目录，页面显示空状态。
4. 输入包含图片的目录，页面显示图片网格。
5. 使用 Steam librarycache 示例路径时，能展示该目录中可读取的图片资源。
